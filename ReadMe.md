## 需要添加等待-通知功能的场景

当前实现的 **"立即返回"** 策略在以下场景下效率低：

| 场景           | 当前行为     | 问题                                 |
| -------------- | ------------ | ------------------------------------ |
| 消费者读空队列 | 返回 `false` | 消费者必须**轮询**（忙等待）浪费 CPU |
| 生产者写满队列 | 返回 `false` | 生产者必须**轮询**等待有空间         |



```
当前模式（轮询 - 低效）：
┌─────────────────────────────────────────────────┐
│ 消费者线程:                                      │
│   while (true) {                                 │
│     if (!queue.pop(val)) {   // 空，返回 false   │
│       // 必须睡眠或继续轮询 ← 浪费 CPU           │
│       std::this_thread::sleep_for(...);         │
│     } else {                                     │
│       process(val);                              │
│     }                                            │
│   }                                              │
└─────────────────────────────────────────────────┘

等待-通知模式（高效）：
┌─────────────────────────────────────────────────┐
│ 消费者线程:                                      │
│   while (true) {                                 │
│     queue.wait_and_pop(val);  // 队列空时自动阻塞 │
│     process(val);                                │
│   }                                              │
│                                                  │
│ 生产者 push 后自动通知等待的消费者 ✓              │
└─────────────────────────────────────────────────┘
```

## 何时需要添加？

| 条件                  | 说明                           |
| --------------------- | ------------------------------ |
| **生产者-消费者模式** | 线程间协作，一个生产、一个消费 |
| **队列可能空/满**     | 不是无界队列，容量有限         |
| **避免忙等待**        | 不想用轮询浪费 CPU             |
| **需要响应及时**      | 数据到达后要立即处理           |

## 典型实现：条件变量

需要添加 `std::condition_variable`：



```cpp
// 伪代码示意
class CircularQueueWithWait {
    std::mutex _mtx;
    std::condition_variable _cv_not_empty;  // 通知"非空"
    std::condition_variable _cv_not_full;   // 通知"非满"
    
    void wait_and_pop(T& val) {
        std::unique_lock lock(_mtx);
        _cv_not_empty.wait(lock, [&] { 
            return _head != _tail;  // 非空才继续 
        });
        // 取数据...
        _cv_not_full.notify_one();  // 通知生产者有空间了
    }
    
    void push(const T& val) {
        std::unique_lock lock(_mtx);
        _cv_not_full.wait(lock, [&] { 
            return (_tail + 1) % _max_size != _head;  // 非满才继续
        });
        // 放数据...
        _cv_not_empty.notify_one();  // 通知消费者有数据了
    }
};
```

什么时候不需要等待-通知功能

## 不需要等待-通知功能的场景

| 场景                    | 说明                           | 示例                              |
| ----------------------- | ------------------------------ | --------------------------------- |
| **1. 轮询模式**         | 需要固定时间间隔检查，不能阻塞 | 游戏循环（60fps）、嵌入式控制循环 |
| **2. 尝试操作**         | 失败就放弃或做别的事，不等待   | 网络包处理（满就丢弃）、日志写入  |
| **3. 批量处理**         | 按批次处理，不需要立即响应     | 定时任务、批处理系统              |
| **4. 超时控制**         | 需要精确控制等待时间           | 带超时的 RPC 调用                 |
| **5. 事件驱动**         | 通过事件循环统一处理           | libuv、Node.js 事件循环           |
| **6. 无界/大容量**      | 队列几乎不会满/空              | 内存充足场景下的缓冲              |
| **7. 生产消费速率匹配** | 队列状态稳定，很少空/满        | 解耦但速率相近的模块              |

## 对比示例



```
场景1：日志系统（不需要等待 - 满则丢弃）
┌─────────────────────────────────────────────────┐
│ if (!log_queue.push(msg)) {                     │
│     // 队列满了，直接丢弃或写入文件              │
│     drop_count++;                               │
│ }                                                │
// 适合：失败可接受，不能阻塞主线程                │
└─────────────────────────────────────────────────┘

场景2：任务调度（需要等待 - 阻塞等待任务）
┌─────────────────────────────────────────────────┐
│ task = task_queue.wait_and_pop();  // 阻塞直到有 │
│ execute(task);                                 │
// 适合：必须处理每个任务，无任务就休眠            │
└─────────────────────────────────────────────────┘

场景3：游戏循环（不需要等待 - 固定帧率）
┌─────────────────────────────────────────────────┐
│ while (running) {                               │
│     process_input_queue();   // 处理完就继续     │
│     update_game_state();                        │
│     render();                                   │
│     sleep_for(16ms);  // 固定 60fps             │
│ }                                                │
// 适合：固定时间步长，不能阻塞                    │
└─────────────────────────────────────────────────┘
```

## 决策树



```
需要等待-通知？
│
├─ 队列会空/满？
│  ├─ 是 → 必须处理每个元素？
│  │  ├─ 是 → 需要 ✓
│  │  └─ 否 → 不需要 ✗（可丢弃/跳过）
│  │
│  └─ 否 → 不需要 ✗（容量充足）
│
└─ 可以阻塞线程？
   ├─ 是 → 可能需要 ✓
   └─ 否 → 不需要 ✗（实时性要求）
```

## 当前 `CircularQueueLk` 的定位

当前设计是**立即返回模式**，适合：

- 需要自己控制轮询/超时逻辑
- 队列作为可选缓冲，失败可接受
- 与外部事件循环集成
- 简单场景，不想引入条件变量复杂度

**结论**：当前实现已经适用于很多场景，等待-通知是**可选增强**，而非必需。
